🔐 synchronized vs ReentrantLock

小区门禁故事 · 并发锁的两种哲学

🚪 老式刷卡门禁
synchronized (内置锁)

  • ✅ 自动开门/关门 (自动加锁/释放锁)
  • ⛔ 不能打断等待 → 干等
  • ⚖️ 非公平锁 (默认不公平，抢着进)
  • 🎭 单一条件 (wait/notify)
  • 📦 可重入 (同一线程可重复进入)
  • 🚫 无尝试锁机制

🏘️ 生活场景

📌 老旧小区大门：你刷卡进入，门自动关上。如果前面有人长时间占着门（比如搬货），你只能在后面干等，不能喊“不等了”。
➕ 优点：简单，无需操心解锁。缺点：不够灵活。

// synchronized 用法
synchronized(lock) {
   // 自动获取锁，退出自动释放
}

📱 智能遥控门禁
ReentrantLock (API锁)

  • ✋ 手动 lock() / unlock() (需finally释放)
  • ⏸️ 可中断 (lockInterruptibly) 等不及可放弃
  • ⚖️ 可选公平锁 (先来后到)
  • 🎭🎭 多条件 (多个Condition，更精细)
  • 📦 可重入 (同一线程可多次lock)
  • 🔍 tryLock() 非阻塞尝试

🏘️ 生活场景

📌 现代智能门禁：你需要按遥控锁门(lock)，走时必须按解锁(unlock)。如果门被占用太久，你可以按“中断”按钮不等了；还可以设置排队顺序(公平锁)，甚至绑定多个信号(比如等快递、等外卖)。
➕ 优点：灵活、强大。缺点：必须记得解锁，否则门永远锁死。

// ReentrantLock 用法
Lock lock = new ReentrantLock();
lock.lock();
try {
   // 临界区
} finally {
   lock.unlock();
}

🔬 小实验：感受“可中断”与“不可中断”

🚪 synchronized 模拟

线程1 占用锁 → 线程2 等待，无法中断，只能傻等。

📱 ReentrantLock 模拟 (可中断)

线程A 持有锁 → 线程B 等待，1秒后打断B，B放弃等待。

※ 演示目的：ReentrantLock 可以响应中断，synchronized 不行。

⚡ 总结：synchronized 简单省心；ReentrantLock 灵活强大（可中断、尝试锁、公平锁、多条件）
